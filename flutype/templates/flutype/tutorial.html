Tutorial: How to upload data

Uploading data into the database consists of two main parts, filling in generic data reused in multiple experiments, and experiment or measurement specific data:

  • The first part is the creation of generic data which is used in multiple measurements or studies.
    This generic data is reused, e.g., ligands like peptides or viruses are only defined once, same for the respective batches (i.e., solutions of peptides, viruses and antibodies), or the experimental steps (so called process steps) which are used in multiple measurements or studies.
    This generic data can be added/modified directly in the browser via the web interface (for instance by adding new peptides or viruses). A user must be logged in to add/modfiy this data.
  • The second part is the upload of data specific for a given measurement/experiment.
    For the upload the information must hereby be provided in a specific directory structure, i.e., certain files with defined names must exist in a given folder.
    When creating the files in this specific directory structure, it is important to check that the generic ligands (peptides, viruses, antibodies, ...), the used batches (solutions of the ligands) and the experimental steps (process steps) have already been uploaded in the database (see generic information above).

The largest unit in the database is a so called study which refers to a collection of measurements. This tutorial leads through the workflow of uploading experimental data for an example study containing measurements from an ELISA binding assay (microwell plate), a peptide-virus binding experiment (microwell plate), and a peptide-virus binding experiment on a peptide-array.

The specific directory structure for this example study is provided in the following supplementary file: flutype-tutorial.zip. The example study was uploaded in the FluTypeDB an is available here. The directory tree of the study can be seen here.

In the following a step by step tutorial to upload the example study is provided:

  • Creating a study
  • Elisa measurements
  • Microwell measurements
  • Microarray measurements

1. Study

The name of the top directory of the tutorial study which we upload is used as a unique identifier sid (identification string) in the FluTypeDB for the study. This must be a unique identifier which is not yet used for another study in the database. In the tutorial the top directory name, and hence sid, is 170929-tutorial.

A study consists off three main building blocks:

  • meta.tsv
  • measurements
  • raw_docs, i.e., additional documents and data files to upload

The metadata defined in the meta.tsv file in the root of the study folder is the obligatory information of a study which describes the key features: what was the reason for the study and what should be answered (description), optional comments (comment),who is responsible for study (user), when was the study started (date),and what is the status of the study (status). metadata fields are hereby:

  • description: text (a description)
  • comment:text (a comment)
  • user : username
  • date: format (YYYY-MM-DD)
  • status: one of the options: planned, in progress and finished
Studies do not have to contain any measurements. One can add a study without any measurements or raw docs by adding a "meta.tsv" into a directory with the name of the study. All of the meta fields are optional, but should be filled out.

Anytime we describe the content of the experimental data directory and which files have to to placed in which locations, have a look at the tutorial example directory where this is demonstrated flutype-tutorial.zip and the corresponding directory tree at directory tree.

Measurements are defined in the measurements folder and can be uploaded for a study at any time, so it is completely legit to upload an empty study at the beginning and fill this step by step with the experimental measurements. At this point we have defined an empty study. How to add measurements, i.e., either ELISA, microarray or microwell measurements to the defined study is described in the following sections.

In addition to measurements for a study it is also possible to upload documents related or important for the study by providing them in raw_docs folder. These can be for instance experimental protocols, analysis, raw data files, etc. All documents of relevance for a study should be uploaded.

2. Elisa Measurement

To add measurements to a study one creates the folder measurements in the study folder. For every measurement an individual subfolder is added

For any measurement, five pieces of information are important:

  • meta.tsv (metadata)
  • lig_fix.txt and lig_mob.txt (ligands)
  • steps.txt (process)
  • results (results)
  • raw_docs (additional documents).
The results and raw_docs are optional.

2.1 Elisa metadata

batch_sid 12345689
measurement_type elisa
functionalization No
manufacturer Thermo F96 Maxisorp
Metadata defined for ELISA measurement in meta.tsv

The batchsid refers to the batch of the microwell plate. The measurement_type has to be chosen from three choices: elisa, microarray and microwell. Both, microarray and microwell refer to fluoresence measurements. The functionalization refers to the surface substance of the microwell plate. In this measurement no ligands were bound to the surface of the microwell plate. Therefore, also no fuctionalization was necessary. We enter a "No", which is different to an empty space. An empty space refers to an unknown value ( NaN ). The manufacturer refers to the manufacturer of the microwell plate. These are the metadata of a measurement. This table is saved as meta.tsv in the main directory of a measurement folder.

2.2 Elisa Ligands

Subsequently, we continue with the next important measurement related information: the ligands and their positions on the microwell plate.

Peptides, antibodies, viruses and any combination of several ligands (complexes) are referred to as ligands. Each of them is an ligand type and includes its specific information. A peptide, for example, is described by a linker, spacer, sequence and c-terminus. Besides the specific information, all ligands share the common fields: sid and comment. However, it might be interesting also to conduct measurements with e.g. different concentrations of the ligands, which is not captured by the ligands related pieces of information. Here, the concept of ligand batches comes into play. A ligand batch links a ligand to the following batch related information: labeling, concentration, buffer, pH, purity and production date. Also for ligand batches, a sid has to be defined. Keep in mind that ligand batch sids are particularly important, because they are later on referenced in the gal files. Ligand batches, can be of five different types depending on what kind of ligand they are linked to: virus batch, peptide batch, antibody batch, complex batch and buffer batch. The buffer batch is a batch without a link to a ligand. The remaining four should be self explaining.

Now back to our measurement. In this example we do not immobilize any ligands on the surface. This reduces our work. No, need to define any peptides or peptide batches. However, still we need a so called gal file, which maps a ligand batch to a location on a microwell plate. Since no ligands are present in any well, we fill all entries in the table with "No". Note that the table has to have the same dimensions as the microwell plate. For microarrays it might be more reasonable to provide a gal file, which is fine too. This table is saved as lig_fix.txt in the main directory of the measurement.

2.3 Elisa Process

The next step is to define the process of our ELISA measurement, i.e., which steps were performed in the experimental workflow. The process is described by process steps within a measurement. A process step can be of various types: blocking, drying, incubating-analyt, incubating, quenching, scanning, spotting and washing. Let me know if you have any suggestions for more. Each step is described by an sid, method, temperature and comment. Some of the steps take also the information substance and duration. Let's take a look at each step.

Once a process step is defined, it can be reused in any other process. Thus, it is part of the general data and is not exclusively related to one measurement. A process can contain many steps in an arbitrary ordering and with additional measurement specific pieces of information. We explain this concept based on our example. So, let's go back to our ELISA example.

Typically, the first step in the process is the spotting/immobilization step. However, no peptides are immobilized in this example. Instead, the first step is the incubation of an antibody. I go quickly through the table above. The first column "step" takes the sid of the step defined before hand. The second column "index" describes the chronological position of the step in the process. The "user" is the person who conducted this step. "Start" is the starting time of the step. The "image" field is left blank for every step in this example. Normally, it takes the name of an image file, which shall be saved in the main directory of the measurement folder. Generally, it is reasonable to use it for microarrays within the scanning step. With all these pieces of information in place, we have complete all the relevant information related to the process. The above table has to be saved as steps.tsv into the main directory of a measurement.

2.4. Elisa Results

In this section, we are going to deal with pieces of information related to the measurement results. Measurement results describe the intensities resulting from the measurement. In this example, we measured the intensities during the last step of our process and saved tem as intensity_final.tsv in the main folder of the measurement. If we want to use these intensities as results we have to save it again in a folder within the results folder. It might seam to be a redundant procedure. However, the aim is to decouple the results from the process. In the microwell example, this approach might become clearer. There, multiple scans have been performed during one process. From these scans different results have been obtained. Often, it might be interesting to subtract the background or to do even more sophisticated post-processing on the measured intensities and to show the output in as results. This concept makes it possible to do these things independently from the measured intensities. Again, the sid of the measurements results are defined by the folder name. This time the folder has to placed into the folder results. We decided to call the result raw. A results folder contains, besides the intensities, also some metadata related to the result. The following table shows the meta.tsv saved in the results folder.

comment results from an elisa intensity reader without any post processing

intensity_file final_intensity.tsv

2.5 Elisa Raw Docs

Just like studies, also measurements can contain additional files (scientific papers, power point presentations, images , ...) , which do not fit into the database structure provided so far. 3.4 Microwell Results

In this example, we want to show how to upload three results. The first result will be the intensities from the the first scanning (process step 7), the second result will be the the intensity from the final scanning and the third will be a subtraction of one from the other. Have a look into the directory 170929-tutorial-microwell-1/results/ in the directory tree. Any directory in the results directory should contain an intensities.tsv and a meta.tsv. The intensities file contains a gal file like intensities. The meta file contains a comment, processing type and intensity_file name. The processing_type describes the method how the intensities were measured (e.g., Florescence Reader, Genepix, Image2Numeric0.1, ...). The comment can be used to described more in detail how the intensity values have been obtained. In the following, the content of meta.tsv of the background subtraction result is shown. The other results can be viewed in the zipped folder.
